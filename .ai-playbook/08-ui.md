# UI & UX Engineering Guide

This document defines engineering rules for designing and implementing user interfaces.

A beautiful UI is not enough.

A successful UI is:

- Understandable
- Accessible
- Predictable
- Fast
- Consistent
- Easy to use

Always optimize for user experience before visual complexity.

---

# Design Philosophy

Users should never think.

Interfaces should feel obvious.

Reduce cognitive load.

Avoid unnecessary interactions.

Every click should have a purpose.

---

# Consistency

Consistency is more valuable than creativity.

Reuse existing patterns.

Avoid inventing new UI behaviors unless they provide measurable improvements.

---

# Visual Hierarchy

Every screen must clearly communicate:

Primary Action

↓

Secondary Action

↓

Supporting Information

Users should immediately know:

- What is important
- What they should do next
- What can be ignored

---

# White Space

Do not fear empty space.

Whitespace improves readability.

Avoid crowded layouts.

---

# Typography

Use typography to communicate importance.

Avoid excessive font sizes.

Avoid using multiple font weights without purpose.

---

# Color

Use color intentionally.

Never rely on color alone to communicate meaning.

Errors, warnings, and success messages must also include icons or text.

---

# Buttons

Buttons should describe actions.

Good

Save Changes

Create User

Download Report

---

Bad

Submit

Click Here

Continue

---

Primary buttons should represent the main action.

Avoid multiple primary buttons in the same section.

---

# Forms

Forms should ask only for necessary information.

Group related fields.

Use meaningful labels.

Never rely only on placeholders.

Always indicate required fields clearly.

Provide inline validation when appropriate.

---

# Inputs

Choose the correct input type.

Email → email

Phone → tel

Date → date

Password → password

Number → number

Do not misuse text inputs.

---

# Validation

Validate as early as possible without becoming intrusive.

Explain errors clearly.

Never show technical error messages to users.

---

Bad

Invalid request

---

Good

Password must contain at least 8 characters.

---

# Tables

Tables are for comparison.

Avoid using tables for card-like content.

Support sorting and filtering only when valuable.

Avoid horizontal scrolling whenever possible.

---

# Cards

Cards should group related information.

Do not overload a single card.

Each card should have one purpose.

---

# Modals

Use modals sparingly.

Do not place long workflows inside modals.

If the interaction becomes complex,

use a dedicated page instead.

---

# Navigation

Navigation should be predictable.

Users should always know:

Where they are.

Where they came from.

Where they can go next.

---

# Breadcrumbs

Use breadcrumbs

for deep navigation.

Avoid breadcrumbs

for shallow navigation.

---

# Loading States

Every asynchronous operation

must communicate progress.

Prefer Skeletons over Spinners.

Skeletons should resemble the final layout.

---

# Empty States

Every empty state should explain:

Why it is empty.

What the user can do next.

Avoid blank screens.

---

# Error States

Every error should:

Explain the problem.

Suggest a solution.

Allow recovery.

Avoid technical jargon.

---

# Success States

Every important action should provide feedback.

Examples:

Toast

Success Banner

Status Badge

Confirmation Screen

---

# Responsive Design

Mobile is not an afterthought.

Every page should work well on:

Mobile

Tablet

Desktop

Ultra-wide screens

Avoid designing only for desktop.

---

# Accessibility

Every interactive element must be keyboard accessible.

Visible focus indicators are required.

Use semantic HTML.

Images require alt text.

Forms require labels.

Never sacrifice accessibility for aesthetics.

---

# Icons

Icons support text.

Icons should not replace text when meaning may be unclear.

Choose consistent iconography throughout the application.

---

# Animations

Animations should improve understanding.

Avoid decorative animations without purpose.

Prefer subtle transitions.

Avoid slowing down the interface.

---

# Notifications

Notifications should be meaningful.

Avoid excessive toasts.

Do not interrupt users unnecessarily.

---

# Search

Search should tolerate minor mistakes.

Support partial matches where appropriate.

Display useful empty results.

---

# Pagination

Use pagination for large datasets.

Use infinite scroll only when it improves the experience.

Administrative systems usually benefit more from pagination.

---

# Destructive Actions

Destructive actions require confirmation.

Delete.

Reset.

Archive.

Remove.

Never make destructive actions too easy to trigger.

---

# Performance

Fast interfaces feel better than visually complex interfaces.

Avoid unnecessary animations.

Avoid layout shifts.

Avoid blocking user interactions.

---

# Design System

Reuse existing components.

Avoid creating new button styles, spacing systems, or typography patterns unless necessary.

Consistency strengthens usability.

---

# AI Design Behavior

When generating UI:

Think like a Product Designer.

Not only a Frontend Developer.

Ask:

Can the user complete the task easily?

Can the interface be simplified?

Can unnecessary elements be removed?

Would a first-time user understand this screen?

---

# Final Checklist

Before generating UI ask:

Is the primary action obvious?

Is the layout consistent?

Is the interface accessible?

Does every state exist?

Loading?

Empty?

Error?

Success?

Is the mobile experience acceptable?

Can anything be removed?

Would this feel professional in a production product?

Only then generate the UI.
